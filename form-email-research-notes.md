# Исследование отправки заявок с WELLS-MO

## Проверенная текущая реализация

Текущая форма в `client/src/pages/SitePages.tsx` не отправляет данные: обработчик `onSubmit` делает только `event.preventDefault()`. В проекте отсутствует рабочий backend-обработчик почты. Файл `server/index.ts` является только статическим сервером для раздачи сборки и маршрутов клиента.

## Проверенная почтовая инфраструктура домена

Домен `wells-mo.ru` имеет рабочие MX-записи `mxs1.reg.ru` и `mxs2.reg.ru`. Также присутствует SPF-запись `v=spf1 ip4:31.31.197.38 a mx include:_spf.hosting.reg.ru ~all`. Это подтверждает, что почта домена настроена на инфраструктуре Reg.ru и адрес `info@wells-mo.ru` может использоваться как адрес получателя.

## Предварительные выводы по сервисам форм

### Web3Forms

Документация Web3Forms прямо позиционирует сервис как решение для статических сайтов без сервера и backend-кода. Основная интеграция строится вокруг публичного `access_key`, который отправляется с формы на API сервиса.

### Formspree

Formspree также прямо поддерживает статические HTML- и JavaScript-формы без серверного кода. Сервис даёт endpoint вида `https://formspree.io/f/{form_id}`, принимает POST-запросы от формы и берёт на себя доставку уведомлений и антиспам-обработку.

## Практическое ограничение

Если сайт остаётся на GitHub Pages как полностью статический, то безопасно скрыть SMTP-пароль или секретный API-ключ внутри фронтенда невозможно. Поэтому надёжные варианты только такие:

1. использовать внешний form backend, где фронтенду выдаётся публичный идентификатор формы;
2. перенести проект на полноценный хостинг с backend и секретами;
3. временно использовать `mailto:` как слабый резервный сценарий, если полноценный сервис ещё не подключён.

## Подтверждённые детали по Web3Forms

Страница установки Web3Forms указывает, что после создания формы пользователь получает `Access Key` по email, а затем вставляет его в скрытое поле формы и отправляет POST-запрос на `https://api.web3forms.com/submit`.

FAQ Web3Forms отдельно уточняет, что `access key` не нужно скрывать: сервис считает его публичным идентификатором, а не секретным API-ключом. При этом риск спама остаётся концептуально похожим на публикацию самого email-адреса, поэтому сервис рекомендует как минимум защиту от спама, а для более жёсткой защиты — captcha и ограничения по домену.

## Практический вывод для WELLS-MO

Для текущего статического сайта на GitHub Pages наиболее реалистичны два рабочих маршрута:

1. **Web3Forms** как быстрый вариант без backend. Его можно внедрить прямо сейчас после того, как владелец почты `info@wells-mo.ru` получит и передаст `access key` либо сам зарегистрирует форму.
2. **Formspree** как более "кабинетный" вариант с собственной панелью и антиспам-функциями, но он тоже требует создания и подтверждения формы у владельца.

Если пользователь не готов сейчас подтверждать внешний сервис через почту `info@wells-mo.ru`, то полноценную автоматическую отправку завершить без его участия нельзя. В этом случае можно временно поставить fallback через `mailto:`, но это будет заметно хуже по конверсии и удобству.

## Результат интерфейсной проверки после внедрения

На странице `/kontakty` форма успешно приняла тестовые данные и после отправки показала зелёный статус успешной доставки: заявка отправлена, обращение получено. После успешной отправки поля формы очистились, что подтверждает прохождение клиентского сценария до положительного ответа API Web3Forms.
